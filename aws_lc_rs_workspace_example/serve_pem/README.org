* serve_pem

Small Axum service that exposes an RSA public key and accepts a hybrid-encrypted
registration payload, then persists registered users into PostgreSQL.

The current code does *not* use Ed25519 anymore. It uses an RSA-2048 keypair to
wrap a per-request AES-256-GCM content-encryption key. Clients encrypt the
registration JSON with AES-256-GCM, encrypt that AES key with the server RSA
public key, and the server reverses that process before hashing the password.

** What The Service Does

- =GET /public-key=
  Returns the server public key in PEM form, the same key in base64 DER, a
  SHA-256 hash of the DER bytes, and the transport metadata clients need for
  RSA-OAEP-256 key wrapping plus AES-256-GCM payload encryption.

- =POST /register=
  Accepts JSON in this format:

#+begin_src json
{
  "wrapped_key_base64": "...",
  "nonce_base64": "...",
  "ciphertext_base64": "..."
}
#+end_src

  The request fields are:

- =wrapped_key_base64=: RSA-OAEP-SHA256 encryption of a random 32-byte AES key
- =nonce_base64=: random 12-byte AES-GCM nonce
- =ciphertext_base64=: AES-256-GCM ciphertext of the registration JSON, with the
  authentication tag appended

  After decryption, the plaintext must be JSON like this:

#+begin_src json
{
  "client_public_key": "client-public-key",
  "password": "super-secret-password"
}
#+end_src

  The server hashes =password= with =Argon2id= before storage. It stores the
  PHC-formatted password hash, never the plaintext password.

- =POST /login=
  Accepts the same hybrid-encrypted request shape as =/register=. After
  decryption, the payload must contain =client_public_key= and =password=. The
  server looks up the user by =client_public_key_sha256= in PostgreSQL and
  verifies the submitted password with =Argon2id=. On failure, it returns
  =401 Unauthorized= with =invalid_credentials=.

** Run

Project layout now looks like this:

#+begin_src text
src/
  main.rs
  lib.rs
  crypto.rs
  db.rs
  error.rs
  handlers.rs
  models.rs
  password.rs
  service.rs
tests/
  login_integration.rs
migrations/
  20260428051919_create_users_table.sql
#+end_src

Default port is =3030=.

Set the database connection string first:

#+begin_src bash
export DATABASE_URL=postgres://postgres:your-password@localhost:5432/serve_pem
#+end_src

Optional but recommended: configure a server-side pepper that is *not* stored
in PostgreSQL:

#+begin_src bash
export PASSWORD_PEPPER='replace-with-a-random-secret-at-least-16-bytes-long'
#+end_src

Create the database and run migrations with =sqlx-cli=:

#+begin_src bash
sqlx database create
sqlx migrate run
#+end_src

#+begin_src bash
cargo run
#+end_src

Run only the integration tests that hit local PostgreSQL:

#+begin_src bash
cargo test --test login_integration -- --nocapture
#+end_src

Run only the unit tests in the library crate:

#+begin_src bash
cargo test --lib
#+end_src

Use a custom port:

#+begin_src bash
PORT=4040 cargo run
#+end_src

When the service starts, it will create or reuse:

- =private_key.pk8= for the RSA private key
- =public_key.der= for the RSA public key

If an old non-RSA =private_key.pk8= exists, the service will back it up as:

#+begin_src text
private_key.pk8.legacy.<timestamp>
#+end_src

and then generate a fresh RSA keypair.

** End-To-End CLI Use Case

This example shows a registration flow using =curl=, =jq=, and =openssl=.

Requirements:

- =curl=
- =jq=
- =openssl=

*** 1. Start The Server

#+begin_src bash
cargo run
#+end_src

Leave that terminal running.

*** 2. Fetch The Server Public Key

#+begin_src bash
curl -s http://localhost:3030/public-key > public-key-response.json
jq -r '.public_key_pem' public-key-response.json > server_public_key.pem
#+end_src

Inspect the metadata if needed:

#+begin_src bash
jq . public-key-response.json
#+end_src

*** 3. Build The Registration Plaintext

Create the JSON that the server expects *after* decryption:

#+begin_src bash
cat > register_payload.json <<'EOF'
{
  "client_public_key": "client-abc-123",
  "password": "correct-horse-battery-staple"
}
EOF
#+end_src

*** 4. Encrypt The Payload With The Hybrid Scheme

The transport is now:

- random 32-byte AES key
- random 12-byte nonce
- AES-256-GCM for the JSON payload
- RSA-OAEP-SHA256 for the AES key

A correct client must:

- fetch =public_key_pem= from =/public-key=
- generate a random 32-byte AES key
- generate a random 12-byte nonce
- encrypt =register_payload.json= with AES-256-GCM
- encrypt the AES key with RSA-OAEP-SHA256 using the server public key
- base64-encode the wrapped key, nonce, and ciphertext

*** 5. Submit =/register=

#+begin_src bash
CIPHERTEXT_BASE64="<aes-gcm-ciphertext-base64>"
WRAPPED_KEY_BASE64="<rsa-wrapped-aes-key-base64>"
NONCE_BASE64="<aes-gcm-nonce-base64>"

curl -s http://localhost:3030/register \
  -H 'content-type: application/json' \
  --data "$(jq -n \
    --arg wrapped_key_base64 "$WRAPPED_KEY_BASE64" \
    --arg nonce_base64 "$NONCE_BASE64" \
    --arg ciphertext_base64 "$CIPHERTEXT_BASE64" \
    '{wrapped_key_base64: $wrapped_key_base64, nonce_base64: $nonce_base64, ciphertext_base64: $ciphertext_base64}')"
#+end_src

Typical response:

#+begin_src json
{
  "status": "registered",
  "user_id": 1,
  "client_public_key_sha256": "..."
}
#+end_src

The server persists the user with a unique =client_public_key_sha256= and an
=Argon2id= password hash string.

Login uses the same request encryption flow and returns:

#+begin_src json
{
  "status": "authenticated",
  "user_id": 1,
  "client_public_key_sha256": "..."
}
#+end_src

** Notes

- The AES key wrapped by RSA must fit inside
  =max_wrapped_key_plaintext_bytes= from =/public-key=. The current service uses
  a 32-byte AES-256-GCM key, which is well within the RSA-OAEP limit.
- If the client uses the wrong RSA padding, wrong nonce length, wrong AES key,
  or tampers with the AES-GCM ciphertext, =/register= will return a decryption
  error.
- The server auto-runs migrations on startup, but keeping =sqlx migrate run= in
  your local setup is still useful for explicit schema management.
