* Overview

This project is an =axum 0.8= example that combines three concerns in one small Rust service:

- custom signup/signin built on =supabase_rs=
- native Supabase Auth signup/signin built on =supabase-auth=
- websocket and optional WSS support built on =axum= websocket APIs

It is useful when you want to compare:

- managing your own user table and password flow
- delegating auth to Supabase GoTrue
- running realtime websocket endpoints in the same Axum service


* What This Project Includes

** 1. Custom Auth via =supabase_rs=

The routes =/auth/signup= and =/auth/signin= use =supabase_rs= as a PostgREST client.

Flow:

- write user records into your own =app_users= table
- hash passwords with =Argon2=
- verify passwords locally
- issue an application token signed with =HMAC-SHA256=

This is the "bring your own auth logic" path.


** 2. Native Auth via =supabase-auth=

The routes =/auth/supabase/signup= and =/auth/supabase/signin= use =supabase-auth=, which talks to Supabase GoTrue.

Flow:

- create users through Supabase Auth
- let Supabase handle password hashing
- let Supabase issue access and refresh tokens
- optionally support email confirmation redirects

This is the "use Supabase Auth as designed" path.


** 3. WebSocket and WSS

The service also exposes:

- =/ws= for websocket upgrade
- =/ws/demo= for a browser demo page

If both =TLS_CERT_PATH= and =TLS_KEY_PATH= are provided, the same app starts with HTTPS and WSS support.


* Why =supabase_rs= and =supabase-auth= Are Not Duplicates

These two crates target different Supabase backends:

| Crate | Backend API | Purpose |
|-------+-------------+---------|
| =supabase_rs= | =/rest/v1/= | database CRUD through PostgREST |
| =supabase-auth= | =/auth/v1/= | authentication through GoTrue |

In this project:

- =supabase_rs= is the data client for your own user table
- =supabase-auth= is the native auth client for Supabase Auth

So the distinction is simple:

- =supabase_rs= does not implement auth for you
- =supabase-auth= does not manage your app table schema for you

| =SUPABASE_JWT_SECRET= | enables =supabase-auth= handlers |

| Variable | Description |

* Recommended Table for Custom Auth

If you want the =/auth/signup= and =/auth/signin= routes to work, create a table like this:

#+begin_src sql
create extension if not exists pgcrypto;

create table public.app_users (
  id uuid primary key default gen_random_uuid(),
  email text not null unique,
  password_hash text not null,
  created_at timestamptz not null default now()
);
#+end_src

* Architecture Notes

** Custom Auth Path

- client sends email and password
- app validates input
- app hashes password with Argon2
- app writes or reads from =app_users= using =supabase_rs=
- app signs a local token with =HMAC-SHA256=

** Supabase Auth Path

- client sends email and password
- app forwards the request through =supabase-auth=
- Supabase GoTrue handles hashing, sessions, and token issuance
- app returns Supabase session data back to the caller

** WebSocket Path

- browser or client upgrades on =/ws=
- server sends a connection event and greeting events
- server and client can exchange JSON or binary messages
- if TLS is enabled, websocket becomes =wss://=


* Good Defaults

If you are choosing between the two auth modes:

- use custom auth if you need full control over schema, token format, and auth logic
- use =supabase-auth= if you want the standard Supabase Auth flow with less custom security code

If you are adding realtime behavior:

- use =/ws= for quick local testing
- use WSS in any browser-facing or production-like environment


* Development Notes

- The service is written for =axum 0.8=
- Static demo assets are served from =/assets=
- WSS is optional, not required for local HTTP development
- Both auth strategies are intentionally kept in one app so the differences are easy to compare
