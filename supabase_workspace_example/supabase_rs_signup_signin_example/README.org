* Overview

This project is an =axum 0.8= example that combines three concerns in one small Rust service:

- custom signup/signin built on =supabase_rs=
- native Supabase Auth signup/signin built on =supabase-auth=
- websocket and authenticated WSS support built on =axum= websocket APIs

It is useful when you want to compare:

- managing your own user table and password flow
- delegating auth to Supabase GoTrue
- running realtime websocket endpoints in the same Axum service


* What This Project Includes

** 1. Custom Auth via =supabase_rs=

The routes =/auth/signup= and =/auth/signin= use =supabase_rs= as a PostgREST client.

Flow:

- write user records into your own =app_users= table
- hash passwords with =Argon2=
- verify passwords locally
- issue an application token signed with =HMAC-SHA256=

This is the "bring your own auth logic" path.


** 2. Native Auth via =supabase-auth=

The routes =/auth/supabase/signup= and =/auth/supabase/signin= use =supabase-auth=, which talks to Supabase GoTrue.

Flow:

- create users through Supabase Auth
- let Supabase handle password hashing
- let Supabase issue access and refresh tokens
- optionally support email confirmation redirects

This is the "use Supabase Auth as designed" path.


** 3. WebSocket and WSS

The service also exposes:

- =/ws= for websocket upgrade
- =/ws/demo= for a browser demo page

If =TLS_CERT_PATH= and =TLS_KEY_PATH= are unset, the app now looks for repo-local dev certs at =certs/localhost.crt= and =certs/localhost.key=. When those files exist, the same app starts with HTTPS and WSS support automatically.


* Why =supabase_rs= and =supabase-auth= Are Not Duplicates

These two crates target different Supabase backends:

| Crate | Backend API | Purpose |
|-------+-------------+---------|
| =supabase_rs= | =/rest/v1/= | database CRUD through PostgREST |
| =supabase-auth= | =/auth/v1/= | authentication through GoTrue |

In this project:

- =supabase_rs= is the data client for your own user table
- =supabase-auth= is the native auth client for Supabase Auth

So the distinction is simple:

- =supabase_rs= does not implement auth for you
- =supabase-auth= does not manage your app table schema for you

* Recommended Table for Custom Auth

If you want the =/auth/signup= and =/auth/signin= routes to work, create a table like this:

#+begin_src sql
create extension if not exists pgcrypto;

create table public.app_users (
  id uuid primary key default gen_random_uuid(),
  email text not null unique,
  password_hash text not null,
  created_at timestamptz not null default now()
);
#+end_src

Example:

#+begin_src bash
export SUPABASE_URL="https://your-project.supabase.co"
export SUPABASE_KEY="your-service-role-or-anon-key"

** Self-Hosted Supabase Local Stack

This repository now includes a local =.env= configured for a Supabase stack running at:

#+begin_src text
http://127.0.0.1:8000
#+end_src

It uses:

- =SUPABASE_KEY= as the local =SERVICE_ROLE_KEY= for PostgREST access
- =SUPABASE_API_KEY= as the local =ANON_KEY= for GoTrue signup/signin
- =SUPABASE_JWT_SECRET= and =AUTH_TOKEN_SECRET= from the local =JWT_SECRET=

If the custom auth table does not exist yet, initialize it with:

#+begin_src bash
/opt/homebrew/opt/postgresql@18/bin/psql \
  postgresql://postgres:$POSTGRES_PASSWORD@127.0.0.1:5432/postgres \
  -f scripts/init-local-supabase.sql
#+end_src

The app also accepts standard self-hosted Supabase variable names as fallbacks:

- =SUPABASE_PUBLIC_URL= or =API_EXTERNAL_URL= for =SUPABASE_URL=
- =SERVICE_ROLE_KEY= or =ANON_KEY= for =SUPABASE_KEY=
- =JWT_SECRET= for =SUPABASE_JWT_SECRET= and =AUTH_TOKEN_SECRET=

If you need a confirmed local Supabase Auth user for =/auth/supabase/signin= and =/ws= testing without mail confirmation flow, create one directly with:

** 3. Verify Axum Endpoints

Run the verification script after the Axum app is up:

#+begin_src bash
./scripts/verify-axum.sh
#+end_src

It checks:

- =GET /health=
- =GET /=
- =GET /ws/demo=
- custom =/auth/signup= and =/auth/signin=

By default it skips native Supabase email auth checks because your self-hosted setup can return =500 Error sending confirmation email= unless the mail service is present. In your Supabase project, =supabase-mail= is defined in:

#+begin_src text
/Users/gerald/test/flutter/supabase-project/dev/docker-compose.dev.yml
#+end_src

To include native Supabase signup:

#+begin_src bash
VERIFY_SUPABASE_SIGNUP=1 ./scripts/verify-axum.sh
#+end_src

To include native Supabase signin after signup succeeds:

#+begin_src bash
VERIFY_SUPABASE_SIGNIN=1 ./scripts/verify-axum.sh
#+end_src

You can also override the base URL:

#+begin_src bash
BASE_URL=https://127.0.0.1:3001 ./scripts/verify-axum.sh
#+end_src

** 3a. Generate Local Dev Certificates

If =certs/localhost.crt= and =certs/localhost.key= are missing, create them once with:

#+begin_src bash
./scripts/generate-certs.sh
#+end_src

After that, =cargo run= will start the server in HTTPS mode automatically unless you explicitly override the TLS paths.

** 4. Verify Auth Handlers

Use the dedicated auth script to exercise the auth endpoints more thoroughly:

#+begin_src bash
./scripts/verify-auth.sh
#+end_src

It checks the custom Axum auth flow for:

- successful =/auth/signup=
- duplicate signup conflict
- successful =/auth/signin=
- wrong-password rejection
- missing-user rejection

Native Supabase email auth remains opt-in because a self-hosted setup without the dev mail service can fail during confirmation-email delivery:

#+begin_src bash

VERIFY_SUPABASE_SIGNIN=1 ./scripts/verify-auth.sh
#+end_src

** Authenticated WebSocket

The websocket endpoint now rejects unauthenticated upgrades. For browsers, pass the Supabase access token as a query parameter. For non-browser clients, you can also send =Authorization: Bearer <token>= during the upgrade request.

Use =/auth/supabase/signin= first, then connect:

#+begin_src bash
SUPABASE_TOKEN="$(curl -k -sS https://127.0.0.1:3000/auth/supabase/signin \
  -H 'content-type: application/json' \
  -d '{"email":"demo@example.com","password":"super-secret-password"}' \
  | jq -r '.access_token')"
#+end_src

Then open:

#+begin_src text
wss://127.0.0.1:3000/ws?access_token=${SUPABASE_TOKEN}
#+end_src

* Architecture Notes

** Custom Auth Path

- client sends email and password
- app validates input
- app hashes password with Argon2
- app writes or reads from =app_users= using =supabase_rs=
- app signs a local token with =HMAC-SHA256=

** Supabase Auth Path

- client sends email and password
- app forwards the request through =supabase-auth=
- Supabase GoTrue handles hashing, sessions, and token issuance
- app returns Supabase session data back to the caller

** WebSocket Path

- browser or client upgrades on =/ws=
- server sends a connection event and greeting events
- server and client can exchange JSON or binary messages
- if TLS is enabled, websocket becomes =wss://=


* Good Defaults

If you are choosing between the two auth modes:

- use custom auth if you need full control over schema, token format, and auth logic
- use =supabase-auth= if you want the standard Supabase Auth flow with less custom security code

If you are adding realtime behavior:

- use =/ws= for quick local testing
- use WSS in any browser-facing or production-like environment


* Development Notes

- The service is written for =axum 0.8=
- Static demo assets are served from =/assets=
- WSS is optional, not required for local HTTP development
- Both auth strategies are intentionally kept in one app so the differences are easy to compare


* Summary

This example is not just a signup/signin demo.

It is a compact reference project showing how to combine:

- =axum 0.8=
- =supabase_rs=
- =supabase-auth=
- websocket / WSS
- custom auth and native Supabase auth in the same service

If you want one project to compare both auth models and also test realtime transport, this repository is that project.
